# 系統異常排查指南

## 查詢入口

```mermaid
flowchart TD
  A[商戶訂單號] --> B[orders]
  B --> C[provider_transactions]
  C --> D[provider_callbacks]
  B --> E[ledger_entries]
  B --> F[merchant_balances]
```

## 常見掉單

```mermaid
flowchart TD
  A[商戶說已付款] --> B{orders 有資料?}
  B -- 否 --> C[查商戶請求紀錄]
  B -- 是 --> D{provider_transactions 有資料?}
  D -- 否 --> E[送三方前失敗]
  D -- 是 --> F{provider_callbacks 有成功?}
  F -- 否 --> G[查 notify 失敗原因]
  F -- 是 --> H{ledger_entries 有入帳?}
  H -- 否 --> I[查入帳流程]
  H -- 是 --> J[查商戶回調]
```

## 查詢欄位

| 情境 | 優先查 |
|---|---|
| 商戶問訂單 | `orders.merchant_order_no` |
| 藍新後台交易 | `provider_transactions.provider_order_no` |
| 藍新交易序號 | `provider_transactions.provider_trade_no` |
| notify 失敗 | `provider_callbacks.error_message` |

### AES padding

```mermaid
flowchart LR
  A[TradeSha 驗證] --> B[PKCS7 解密]
  B -->|失敗| C[零填充相容解析]
  C --> D[驗證訂單與金額]
```

實測正式環境：首次通知可能是無 PKCS7 padding 的 AES/CBC，後台補發則可用標準 PKCS7。系統以 PKCS7 為主，僅在 `TradeSha` 正確且明文格式有效時接受零填充或無填充。

驗證失敗仍會寫入 `provider_callbacks`。日誌只記錄密文長度與 SHA256 指紋，用來比較自動 Retry 和後台補發，不記錄通知明文。

建立交易時固定傳送 `EncryptType=0`，不可只依賴藍新預設值。
| 金額不符 | `orders.amount_cents`、`provider_transactions.amount_cents` |
| 未入帳 | `ledger_entries.order_id` |

## 狀態判斷

```mermaid
flowchart LR
  A[pending] --> B[尚未收到成功通知]
  C[paid] --> D[已付款並入帳]
  E[failed] --> F[付款失敗]
  G[expired] --> H[逾期未付款]
```

## 必留資料

| 表 | 必留原因 |
|---|---|
| `orders` | 平台主訂單 |
| `provider_transactions` | 對應三方交易 |
| `provider_callbacks` | 保存原始 notify |
| `ledger_entries` | 對帳與餘額依據 |

## 查單流程

```mermaid
sequenceDiagram

  Ops->>PSP: 比對三方後台
  Ops->>DB: 查 ledger_entries
```
