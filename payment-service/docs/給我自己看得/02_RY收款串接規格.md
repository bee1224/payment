# 收款串接規格

> 最後更新：2026-07-11
> 歷史檔名保留 `RY`，但本文件目前描述的是 `RIG001 Gateway` 對外收款契約

## 正式接口

- `POST /api/pay_order`
- `POST /api/query_transaction`

相容路由 `POST /api/v1/deposits` 與 `POST /api/v1/deposits/query` 仍存在，但已標記 deprecated。

## 收款渠道對照

| `pay_channel_id` | `channel_code` | 說明 |
|---|---|---|
| `1000` | `CREDIT` | 信用卡 |
| `1001` | `APPLEPAY` | Apple Pay |
| `1002` | `GOOGLEPAY` | Google Pay |
| `1005` | `WEBATM` | WebATM |
| `1006` | `VACC` | ATM 虛擬帳號 |
| `1007` | `CVS` | 超商代碼 |
| `1008` | `BARCODE` | 超商條碼 |

## 建單重點

- 商戶號欄位是 `pay_customer_id`
- 目前正式商戶號是 `RIG001`
- `pay_md5_sign` 由我方 gateway 相容規則驗證
- provider 真正出款/收款通道目前可對接 NewebPay，但商戶端不直接面對藍新欄位

## 查單重點

- `pay_order_id` 可單筆，也可陣列
- 回應狀態目前主要使用：
  - `0`：處理中或待付款
  - `2`：已付款
  - `5`：失敗

## 商戶 callback

- provider 通知入站後，服務會將整理過的 gateway 相容 payload POST 到 `pay_notify_url`
- 商戶端需回純文字 `OK`
- callback `status` 對照：
  - `30000`：paid
  - `50000`：failed
  - `10000`：processing

## 對外語意提醒

- 對商戶說明時，不要再把三方密鑰說成藍新 `HashKey` / `HashIV`
- 商戶使用的是：
  - 三方商戶號：`RIG001`
  - 三方密鑰：我方發放的 API Key
