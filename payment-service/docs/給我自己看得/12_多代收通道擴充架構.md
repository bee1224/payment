# 多代收通道擴充架構

這份文件整理目前專案已預留的多代收擴充骨架，以及未來接入新代收供應商（例如客樂得）時的落點。

## 目前已完成的骨架

### 1. Provider registry

- `DepositService` 已從單一 `gateway` 改為多個 `gateways`
- 目前可依 `provider_code` 或 `channel_code -> provider_code` 映射選擇代收供應商

主要位置：

- [internal/service/deposit_service.go](../internal/service/deposit_service.go)
- [internal/app/server.go](../internal/app/server.go)

### 2. 訂單不再寫死單一 provider

- 建單時 `order.Provider` 已不再固定為 `newebpay`
- 可由 `provider_code` 或通道映射決定

主要位置：

- [internal/service/deposit_service.go](../internal/service/deposit_service.go)

### 3. 通知流程改為可支援多 provider

- 正式 generic notify route：`POST /api/v1/deposits/providers/{provider}/notifications`
- `POST /notify/{provider}` 與 `POST /notify/newebpay` 僅保留相容
- `HandleDepositProviderNotification(providerCode, fields)` 已可依 provider 分流

主要位置：

- [internal/delivery/http/router.go](../internal/delivery/http/router.go)
- [internal/delivery/http/notify_handler.go](../internal/delivery/http/notify_handler.go)
- [internal/service/deposit_service.go](../internal/service/deposit_service.go)

### 4. Repository / DB provider 關聯已打開

- `DepositStore` 的通知處理已改為帶 `providerCode`
- 建單與通知處理會依訂單 / 通知所屬 provider 寫入與查詢
- `payment_channels` 初始 migration 已改為 `(provider_id, code)` 唯一鍵，避免不同 provider 不能共用同名通道代碼

主要位置：

- [internal/repository/deposit_store.go](../internal/repository/deposit_store.go)
- [migrations/001_init.sql](../migrations/001_init.sql)

## 未來接入客樂得時要補的點

### 1. 新增 provider 實作

參考 `newebpay` 目錄，新建例如：

- `internal/provider/kld/client.go`
- `internal/provider/kld/notify.go`
- `internal/provider/kld/mapper.go`

需實作介面：

- `CreatePayment(order, itemDesc)`
- `VerifyNotification(fields)`

介面位置：

- [internal/provider/gateway.go](../internal/provider/gateway.go)

### 2. 在 server 註冊客樂得 gateway

需在啟動時加入：

- `gateways["kld"] = ...`
- `channelProviders["某些通道"] = "kld"`

位置：

- [internal/app/server.go](../internal/app/server.go)

### 3. 補客樂得 config

目前 `config` 只有 `newebpay` 與 `ry`，未來需新增例如：

- `KLDConfig`
- `KLD_*` env

位置：

- [internal/config/config.go](../internal/config/config.go)

### 4. 補客樂得 notify URL

正式通知路徑：

- `POST /api/v1/deposits/providers/kld/notifications`

只要 `VerifyNotification` 實作好，通知流程就可掛上。

## 目前仍未做的更進一步能力

- 依商戶維度選 provider
- 依通道優先序 / 健康度自動路由
- provider fallback
- callback retry / outbox
- provider response payload 標準化更細分
