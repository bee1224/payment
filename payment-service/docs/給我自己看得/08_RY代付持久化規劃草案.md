# RY 代付持久化規劃草案

> 範圍只包含代付。商戶、Provider、商戶餘額與總帳沿用既有資料；代收 `orders` 與 `provider_transactions` 不納入本圖。
>
> **本文件是未實作草案。** `migrations/001_init.sql` 目前沒有本文的 `payout_*` 資料表；現行程式只完成 RY HTTP client、代理端點與回調驗簽。

### 重要唯一鍵

- `payout_orders (merchant_id, merchant_payout_no)`：商戶端冪等，避免 API 重送造成重複出款。
- `payout_transactions (payout_order_id, attempt_no)`：每次送 RY 都是獨立 attempt。
- `payout_transactions (provider_id, provider_order_no)`：Provider 訂單不可重複。
- `payout_callbacks (provider_id, provider_event_id)`：若 RY 沒有事件編號，改用已正規化 payload 的雜湊去重。
- `ledger_entries.entry_no`：每個會計事件只能入帳一次。

`payout_beneficiaries` 是下單當下的不可變快照，不能只指向可編輯的常用收款人資料，否則歷史訂單會被後續修改污染。完整帳號、戶名與身分資料需加密保存，log、request/response 與 callback 僅留遮罩值。

## 代付資料流

```mermaid
sequenceDiagram
  autonumber
  participant M as 商戶
  participant API as Payment API
  participant DB as MySQL
  participant W as Payout Worker
  participant N as RY 代付
  participant R as Reconciliation Worker

  M->>API: POST /payouts（含 merchant_payout_no）
  API->>API: 驗證商戶、額度、收款資料與冪等鍵
  API->>DB: BEGIN + 鎖定 merchant_balances
  API->>DB: 建立 payout_order 與 beneficiary 快照
  API->>DB: available -= total_debit；pending += total_debit
  API->>DB: 寫入 payout_hold ledger
  API->>DB: COMMIT
  API-->>M: payout_no + holding

  W->>DB: 鎖定一筆 holding/retryable 訂單
  W->>DB: 建立 payout_transaction attempt
  W->>N: POST /api/payments/pay_order
  N-->>W: 受理或同步拒絕
  W->>DB: 保存遮罩後回應，更新 submitted/failed

  alt RY 非同步通知
    N->>API: payout callback
    API->>DB: 先保存 callback 原文遮罩與去重鍵
    API->>API: 驗章、解密、比對金額與訂單
    API->>DB: 原子更新 transaction/order/ledger/balance
    API-->>N: ACK
  else 通知遺失或長時間 processing
    R->>N: POST /api/payments/query_transaction
    N-->>R: 最新狀態
    R->>DB: 使用同一套結果處理器更新
  end

  API-->>M: 商戶 callback（最終狀態）
  M->>API: GET /payouts/{payout_no}（可補查）
```

## 餘額與總帳資料流

```mermaid
flowchart TD
  A[建立代付] --> B{available 是否足夠}
  B -- 否 --> X[拒絕，不建立出款]
  B -- 是 --> C[凍結 total_debit]
  C --> C1[available 減少]
  C --> C2[pending 增加]
  C --> D[送 RY]
  D --> E{最終結果}
  E -- 成功 --> F[pending 減少]
  F --> F1[寫 payout_completed ledger]
  E -- 拒絕或失敗 --> G[解除全額凍結]
  G --> G1[pending 減少]
  G --> G2[available 增加]
  G --> G3[寫 payout_released ledger]
  E -- 已成功後退匯 --> H[依 RY 實際退回金額回補]
  H --> H1[寫 payout_returned ledger]
```

若未來實作本草案，餘額異動、訂單狀態與 `ledger_entries` 必須在同一個資料庫交易內完成，並鎖定該商戶幣別的 `merchant_balances`。代付金額與手續費一開始一起凍結；退匯時手續費是否退回，依 RY 契約的實際結果入帳，不能先假設全退。

## 建議狀態

```mermaid
stateDiagram-v2
  [*] --> holding
  holding --> reviewing: 需人工或風控覆核
  reviewing --> holding: 核准
  reviewing --> rejected: 拒絕
  holding --> submitting
  submitting --> processing: RY 已受理
  submitting --> failed: 同步拒絕且不可重試
  submitting --> holding: 技術錯誤，可安全重試
  processing --> completed: 出款成功
  processing --> failed: 出款失敗
  completed --> returned: 銀行退匯
  holding --> cancelled: 送件前取消
```

只有 `completed`、`failed`、`rejected`、`cancelled`、`returned` 是規劃中的業務終態；網路逾時不是失敗，必須先查 RY 狀態，避免重送造成雙重出款。
