# RY 代付串接規格

> 文件定位：描述目前程式裡「代付」實際已落地的能力，包含 RY 代理層與我方本地提款工作流。
>
> 最後對齊：2026-07-07

## 一句話先講清楚

代付現在分成兩層：

1. `/api/payments/*`：直接對 RY 的 provider contract 代理
2. `/api/payouts/*`：我方正式提款流程 API，會落地本地訂單、凍結額度、審核、送單、補查、接回調、通知商戶

## 已實作端點

### A. RY 代理層

| 方法 | 我方端點 | RY 上游端點 | Function |
|---|---|---|---|
| `POST` | `/api/payments/pay_order` | `/api/payments/pay_order` | `CreatePayoutOrder` |
| `POST` | `/api/payments/query_transaction` | `/api/payments/query_transaction` | `QueryPayoutTransaction` |
| `POST` | `/api/payments/balance` | `/api/payments/balance` | `QueryPayoutBalance` |
| `POST` | `/api/payments/callback` | 由 `RY_PAYOUT_NOTIFY_URL` 指定 | `PayoutCallback` |

### B. 我方提款工作流

| 方法 | 我方端點 | Function | 用途 |
|---|---|---|---|
| `POST` | `/api/payouts` | `CreateWorkflowPayout` | 建立提款申請 |
| `POST` | `/api/payouts/query` | `QueryWorkflowPayout` | 依商戶單號或我方單號查詢 |
| `GET` | `/api/payouts/{payout_no}` | `GetWorkflowPayout` | 依我方單號查詢 |
| `POST` | `/api/payouts/{payout_no}/approve` | `ApproveWorkflowPayout` | 審核通過並送 RY |
| `POST` | `/api/payouts/{payout_no}/reject` | `RejectWorkflowPayout` | 審核拒絕並釋放額度 |
| `POST` | `/api/payouts/{payout_no}/cancel` | `CancelWorkflowPayout` | 在安全條件下取消提款單 |

## 本地工作流現況

目前程式已落地以下資料與流程：

- `payout_orders`
- `payout_beneficiaries`
- `payout_transactions`
- `payout_callbacks`
- `merchant_payout_callback_tasks`
- `merchant_balances`
- `ledger_entries`

目前狀態包含：

- `pending_review`
- `approved`
- `submitting`
- `processing`
- `completed`
- `failed`
- `reversed`
- `rejected`
- `cancelled`

## 建立提款申請 `/api/payouts`

### 必要欄位

- `merchant_id`
- `api_key`
- `merchant_payout_no`
- `amount`
- `pay_account_name`
- `pay_card_no`
- `pay_bank_name`

### 目前驗證規則

- `amount` 必須大於 0
- `currency` 預設 `TWD`
- `pay_bank_name` 必須為 3 碼
- `pay_bank_name` 必須落在 RY 銀行代碼白名單中
- 相同商戶 `merchant_payout_no` 具冪等性，不會重複建單
- 建單時會先凍結商戶可用餘額

### `callback_url` 的意思

`/api/payouts` 內的 `callback_url` 是 **我方通知商戶提款結果** 的網址。  
它不是 RY 回打我方的網址。

RY 回打我方使用的是環境變數：

- `RY_PAYOUT_NOTIFY_URL`

## 審核 / 取消權限

以下端點需要審核權限：

- `POST /api/payouts/{payout_no}/approve`
- `POST /api/payouts/{payout_no}/reject`
- `POST /api/payouts/{payout_no}/cancel`

授權方式：

- `X-Payout-Review-Token: <token>`
- 或 `Authorization: Bearer <token>`

對應環境變數：

- `PAYOUT_REVIEW_TOKEN`

## 送單到 RY 的行為

- 審核通過後，系統會以 `merchant_payout_no` 作為 `pay_order_id` 送往 RY
- `pay_notify_url` 一律使用 `RY_PAYOUT_NOTIFY_URL`
- 我方會重算 `pay_md5_sign`
- `RY_CUSTOMER_ID`、`RY_SIGN_KEY` 由系統設定補入
- 若同步送單成功，訂單進入 `processing`
- 若同步送單失敗，系統會保留 attempt 與錯誤資訊，並依錯誤型態決定是否可補查 / 重判

## 查單與補查

- `GET /api/payouts/{payout_no}` 與 `POST /api/payouts/query` 在查到 `approved` / `submitting` / `processing` 時，會嘗試向 RY 補查最新狀態
- timeout 後不應直接重送 RY，應優先查單

## RY 回調

### 驗證規則

- 驗簽時排除 `sign`
- 目前接受：
  - `transaction_code=30000`：成功
  - `transaction_code=40000`：失敗 / 沖正
- 驗證成功後回應純文字 `OK`

### 回調處理結果

- 寫入 `payout_callbacks`
- 以 provider event key 去重
- 更新本地訂單狀態
- 更新餘額與帳務
- 若進入終態，建立商戶回調任務

## 商戶回調

當提款單進入終態時，若訂單有 `callback_url`，系統會 POST JSON 通知商戶。

目前通知欄位包含：

- `merchant_id`
- `merchant_payout_no`
- `payout_no`
- `provider_order_no`
- `status`
- `amount`
- `fee`
- `currency`
- `completed_at`
- `sign`

目前有 outbox / retry 機制：

- 會建立 `merchant_payout_callback_tasks`
- 成功條件為 HTTP 2xx 且回應本文為 `OK`
- 失敗後會依退避時間重試

## 取消規則

`POST /api/payouts/{payout_no}/cancel` 並不是任意狀態都能取消。

目前僅允許：

- `pending_review`
- `approved` 且尚未真正送上游的安全狀態

以下情況不可取消：

- 已有明確送單 attempt
- 已有 `submitted_at`
- 已有上游單號 / 交易號
- 已完成、失敗、沖正、拒絕等終態

所以「已付款 / 已出款成功」的訂單，不會被這支取消 API 影響。

## 必要設定

| 環境變數 | 用途 |
|---|---|
| `RY_BASE_URL` | RY 上游站台 Base URL |
| `RY_CUSTOMER_ID` | 上游商戶 ID |
| `RY_SIGN_KEY` | MD5 簽名密鑰 |
| `RY_PAYOUT_NOTIFY_URL` | RY 回打我方的網址 |
| `RY_HTTP_TIMEOUT_SECONDS` | 上游 HTTP 逾時秒數 |
| `PAYOUT_REVIEW_TOKEN` | approve / reject / cancel 權限驗證 |
