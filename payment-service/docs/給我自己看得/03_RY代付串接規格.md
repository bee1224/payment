# 代付串接規格

> 最後更新：2026-07-11
> 歷史檔名保留 `RY`，但本文件目前描述的是 `RIG001 Gateway` 現況

## 兩層接口

### 1. Provider 相容代理層

- `POST /api/payments/pay_order`
- `POST /api/payments/query_transaction`
- `POST /api/payments/balance`
- `POST /api/payments/callback`

用途：
- 對上游相容
- 除錯 provider contract
- 驗證上游簽名與回調

### 2. 我方正式 workflow API

- `POST /api/payouts`
- `POST /api/payouts/query`
- `GET /api/payouts/{payout_no}`
- `POST /api/payouts/{payout_no}/approve`
- `POST /api/payouts/{payout_no}/cancel`
- `POST /api/payouts/{payout_no}/resend-callback`

## 正式 workflow 重點

- 商戶建單使用：
  - `merchant_id`
  - `api_key`
  - `merchant_payout_no`
  - `amount`
  - `pay_account_name`
  - `pay_card_no`
  - `pay_bank_name`
- `merchant_payout_no` 為商戶冪等鍵
- `pay_bank_name` 需為三碼銀行代碼，且需在白名單內
- callback 成功後會推送至商戶 `callback_url`

## 審核與安全

- `approve` / `cancel` / merchant key 管理接口都需要 review token
- 使用 header：
  - `X-Payout-Review-Token: <token>`

## 已完成能力

- 本地 payout order / beneficiary / transaction 持久化
- provider callback 驗簽
- 商戶 callback outbox / retry
- callback resend
- merchant API key hash 驗證
- merchant API key list / rotate / revoke

## 仍需注意

- `/api/payments/*` 是相容層，不是建議商戶正式長期依賴的 workflow 契約
- 商戶正式對接優先走 `/api/payouts/*`

## 環境變數

| 變數 | 用途 |
|---|---|
| `GATEWAY_BASE_URL` | 上游 gateway base URL |
| `GATEWAY_CUSTOMER_ID` | 上游 customer id |
| `GATEWAY_SIGN_KEY` | 上游簽名 key |
| `GATEWAY_PAYOUT_NOTIFY_URL` | provider 回我方的 callback URL |
| `GATEWAY_HTTP_TIMEOUT_SECONDS` | 上游 HTTP timeout |
| `PAYOUT_REVIEW_TOKEN` | 審核操作 token |
