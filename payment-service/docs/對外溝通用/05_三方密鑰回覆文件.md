# RIG001 Gateway 三方密鑰回覆文件

> 文件版本：2026-07-11
> 適用對象：合作方 / 上游 / 對接窗口

本文件專門用於回覆合作方關於「三方商戶號」與「三方密鑰」的定義、發放方式與保存方式。

## 一、回覆重點

1. 我方並非要求合作方直接對接藍新。
2. 我方已自行建立一層自有 `gateway`，對外識別名稱為 `RIG001 Gateway`。
3. 合作方對接時使用的三方商戶號為 `RIG001`。
4. 三方密鑰由我方 `gateway` 自行發放與管理，不是藍新的 `HashKey` / `HashIV`。
5. 三方密鑰不是由藍新的參數、商戶號或交易資料再經 SHA-256 推導出來。
6. 合作方實際持有的是我方發放的原始 API Key；我方系統內部資料庫只保存其 SHA-256 hash 值，不保存可逆明碼。

## 二、正式說明

### 1. 三方商戶號

合作方接入我方自建 `gateway` 時，請使用以下三方商戶號：

`RIG001`

此商戶號代表合作方是對接我方 `RIG001 Gateway`，不是直接對接藍新原始通道。

### 2. 三方密鑰

三方密鑰由我方 `gateway` 自行建立、發放、輪替與失效管理。

合作方使用的三方密鑰：

- 是我方發放的 API Key
- 不是藍新的 `HashKey`
- 不是藍新的 `HashIV`
- 不是把藍新參數或商戶號再做 SHA-256 後得到的值

### 3. SHA-256 的用途

SHA-256 只用於我方系統內部的安全化存放。

也就是：

- 合作方拿到的是原始 API Key
- 我方在寫入資料庫時，會將該 API Key 做 SHA-256 hash 後保存
- 我方在驗證時，會將合作方送來的 API Key 再做同樣的 hash 比對

因此，SHA-256 是「保存與驗證機制」的一部分，不是「對外密鑰生成規則」。

## 三、避免誤解

以下理解是錯的：

1. 「三方密鑰 = 藍新 HashKey」
2. 「三方密鑰 = 藍新 HashIV」
3. 「三方密鑰 = 把藍新商戶號、HashKey、HashIV 或交易參數做 SHA-256 算出來」

正確理解是：

1. 藍新憑證是我方對接上游通道時使用
2. 三方商戶號與三方密鑰是合作方對接我方 `gateway` 時使用
3. 兩者屬於不同層級，不應混用

```text
另補充說明，三方密鑰不是藍新的 HashKey / HashIV，也不是將藍新相關參數再經 SHA-256 推導而成。

```
